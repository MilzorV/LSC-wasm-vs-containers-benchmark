# Three-week Project Plan - Upstream Meilisearch on Spin/wasmtime vs OCI

This plan implements [requirements.md](requirements.md). The project now prioritizes an honest upstream-port feasibility study: use the same pinned Meilisearch version on both sides where possible, run the official OCI container as the native baseline, and measure cold start, memory isolation/overhead, and throughput on a shared API surface.

## Stack fixed for the project

| Layer | Choices |
|---|---|
| Upstream source | `meilisearch/meilisearch` tag `v1.43.0`, fetched into `vendor/meilisearch/` by script |
| OCI baseline | Official `getmeili/meilisearch:v1.43.0` through Docker Compose |
| Spin workload | Rust Spin HTTP component targeting `wasm32-wasip2` |
| Porting workspace | `spin-meili/crates/spin-http-adapter`, `spin-meili/crates/meili-wasi-compat`, and legacy subset crates as fallback/reference only |
| Runtime | Spin CLI `3.6.3`; wasmtime through Spin |
| Native checks | Upstream cargo check on host using Meilisearch's pinned Rust toolchain |
| WASI checks | Layered `cargo check --target wasm32-wasip2` for selected upstream crates |
| Benchmarks | Bash orchestration, Python load generation/analysis, raw CSV outputs |
| Analysis | Python tables and plots under `results/processed/` and `results/plots/` |
| Docs | `requirements.md`, methodology, environment record, consultation notes, final report |

**Ports:** Spin on `127.0.0.1:8080`; OCI Meilisearch on `127.0.0.1:8081`.

**Pinned versions:** upstream source tag `v1.43.0`, observed commit `475ed56e5612df0dbb826748add5f93e0e7d5500`, OCI image `getmeili/meilisearch:v1.43.0`.

## Week 1 - Source pinning, smoke parity, and feasibility evidence

- Pin the upstream source and OCI baseline:
  - fetch `meilisearch/meilisearch` tag `v1.43.0`;
  - record the exact commit;
  - keep OCI Compose pinned to `getmeili/meilisearch:v1.43.0`;
  - document the source/image relationship.
- Keep the existing Spin subset running as a fallback/reference:
  - rename the custom search crate to make the fallback status explicit;
  - keep the Spin HTTP surface available for smoke tests;
  - keep fixture and smoke scripts unchanged where possible.
- Add porting workspace pieces:
  - `spin-meili/crates/meili-wasi-compat` for WASI-specific compatibility boundaries;
  - `spin-meili/crates/spin-http-adapter` as the only Spin HTTP server layer;
  - docs that explain the intended boundary between Spin and upstream Meilisearch logic.
- Add feasibility scripts:
  - `scripts/fetch-meilisearch.sh`;
  - `scripts/check-upstream-native.sh`;
  - `scripts/check-upstream-wasi.sh`.
- Run the native upstream check:
  - verify that the pinned source compiles/checks on the host;
  - save logs outside the tracked source files.
- Run layered WASI checks:
  - start with small reusable crates;
  - attempt `milli`, `meilisearch-types`, `routes`, and `meilisearch`;
  - generate `docs/upstream-wasi-blockers.md`.
- Maintain smoke parity:
  - Spin `/health`, fixture load, and `space` search;
  - OCI `/health`, fixture load with Bearer auth, task wait, and `space` search.

**Exit:** upstream source is pinned, native check succeeds, WASI blocker evidence is documented, OCI smoke works, and the Spin fallback remains usable while the upstream port boundary is investigated.

## Week 2 - Porting boundary and benchmark harness

- Analyze WASI blockers:
  - separate immediate build blockers from deeper runtime/storage blockers;
  - identify the highest upstream Meilisearch crates that can be reused;
  - decide whether to patch, feature-gate, or wrap each blocker.
- Attempt source-sharing integration in Spin:
  - reuse upstream request/API types if they compile;
  - reuse upstream filter/query parsing where possible;
  - attempt `milli` integration only after dependency blockers are understood;
  - keep all storage/runtime substitutions in `meili-wasi-compat`.
- Decide the benchmark implementation tier:
  - full upstream logic through Spin;
  - upstream API/types plus compatibility storage;
  - legacy subset fallback with documented blocker evidence.
- Implement benchmark scripts:
  - cold start campaign;
  - throughput/latency campaign;
  - memory sampling;
  - common fixture setup and reset helpers;
  - raw CSV schemas with Git SHA, timestamp, system, route, concurrency, latency, throughput, errors, and memory fields.
- Run pilot benchmark campaigns:
  - same fixture for both systems;
  - same index UID and primary key;
  - same ingestion/search request JSON where possible;
  - concurrency levels `10`, `50`, `100`, `200`.

**Exit:** the project has a chosen Spin implementation tier, benchmark scripts produce raw CSVs for both systems, and all deviations from native Meilisearch are recorded.

## Week 3 - Results, interpretation, and report

- Process CSVs into tables and plots:
  - cold-start distribution;
  - throughput versus concurrency;
  - p95/p99 latency;
  - memory by phase;
  - error rates if any.
- Write the final report:
  - assigned topic and project goal;
  - pinned source/image versions;
  - upstream native build evidence;
  - WASI blocker evidence;
  - selected Spin implementation tier;
  - benchmark methodology;
  - cold-start results;
  - memory/isolation comparison;
  - throughput/latency results;
  - limitations and reproducibility steps;
  - final recommendation about Wasm as a serverless isolation substrate.
- Make limitations explicit:
  - whether the Spin service is full upstream, partial upstream, or fallback subset;
  - which Meilisearch layers could not be reused;
  - whether LMDB/mmap storage was replaced;
  - why memory measurements differ between Spin and OCI;
  - how much confidence the results support.
- Prepare consultation/demo notes:
  - show native upstream check;
  - show WASI blocker report;
  - run Spin and OCI smoke checks;
  - show one benchmark output and one plot;
  - state the recommendation in one clear sentence.

**Exit:** submission-ready evidence, reproducible benchmark outputs, honest limitations, and a clear conclusion.
